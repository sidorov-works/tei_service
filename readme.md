## `kb-tool`: Инструмент управления базой знаний (RAG)

`kb-tool` — это универсальная командная утилита для подготовки, индексации и управления документами в векторной базе данных (Qdrant), которая служит основой для RAG-системы ИИ-ассистента технической поддержки.

Инструмент реализует полный ETL-цикл: от сырых файлов (DOCX, HTML) до семантических чанков, готовых к поиску.

### Основные возможности

*   **Конвертация (`convert`)**: Преобразование `.docx` и `.html` в чистый Markdown (`.md`).
*   **Чанкинг (`chunk`)**: Интеллектуальное разбиение документов на логические фрагменты (чанки) с помощью LLM (DeepSeek) и автоматической нарезкой больших текстов.
*   **Индексация (`index`)**: Загрузка чанков в Qdrant с гибкой системой тегирования, извлекаемой из имени файла.
*   **Поиск (`search`)**: Тестовый поиск по векторной БД для проверки качества индексации.
*   **Управление (`delete`, `list`)**: Просмотр, удаление документов и статистика по тегам.

### Пользовательская документация

#### 1. Установка

1.  Клонируйте репозиторий проекта и перейдите в директорию `doc_proc/` (где находится `kb-tool`).
2.  Создайте и активируйте виртуальное окружение:
    ```bash
    python -m venv venv
    source venv/bin/activate  # Linux/Mac
    # venv\Scripts\activate  # Windows
    ```
3.  Установите пакет в режиме разработки (это установит все зависимости, включая `mammoth` для DOCX):
    ```bash
    pip install -e .
    ```

#### 2. Конфигурация

Перед началом работы необходимо настроить подключения к внешним сервисам. Скопируйте пример файла `.env` и отредактируйте его:

```bash
cp .env.example .env
# Отредактируйте .env, указав ваши ключи и адреса
```

**Ключевые переменные:**

*   `INTERNAL_API_SECRET`: Секретный ключ для внутренних API.
*   `SELF_REDIS_*`: Настройки подключения к Redis (используется как каталог товаров).
*   `ENCODER_SERVICE_URL`: Адрес сервиса эмбеддингов.
*   `DEEPSEEK_APIKEY`: Ваш API-ключ DeepSeek.
*   `QDRANT_HOST`: Адрес хоста с запущенным Qdrant.
*   `DEFAULT_TENANT`: Тенант, который будет использоваться, если не указан флаг `--tenant`.

#### 3. Работа с документами (ETL)

Основной рабочий процесс состоит из трех последовательных шагов.

##### Шаг 1: Конвертация в Markdown (`convert`)

Преобразует исходные файлы в единый формат `.md`.

*   **Поддерживаемые форматы:** `.docx`, `.html`, `.htm`.
*   **Для чего:** Удаляет "мусорный" HTML-код, извлекает чистый текст, конвертирует Word-документы.

```bash
# Конвертировать один файл
kb-tool convert инструкция.docx

# Конвертировать все поддерживаемые файлы в папке (пакетный режим)
kb-tool convert ./raw_docs/ --batch

# Указать выходную папку для результатов
kb-tool convert ./raw_docs/ --batch -o ./converted_md/
```

##### Шаг 2: Чанкинг (`chunk`)

Разбивает Markdown-файл на семантические чанки. На этом этапе текст отправляется в LLM, которая выделяет логические блоки и присваивает каждому чанку темы (`topics`).

*   **Вход:** `.md` файл.
*   **Выход:** `.json` файл со структурой, готовой для индексации.
*   **Автоматическая нарезка:** Если файл слишком большой для LLM (больше `MAX_CHUNKER_INPUT_SIZE`), он автоматически нарезается на части с перекрытием (`CHUNK_OVERLAP_SIZE`), чтобы не потерять контекст на стыках.

```bash
# Разбить один файл
kb-tool chunk инструкция.md

# Пакетная обработка всех .md файлов в папке
kb-tool chunk ./converted_md/ --batch

# Сохранить JSON-файлы в отдельную папку
kb-tool chunk ./converted_md/ --batch -o ./chunks_json/
```

##### Шаг 3: Индексация (`index`)

Загружает JSON-файлы с чанками в векторную базу данных. **На этом этапе критически важно правильно назвать файл**, так как из его имени извлекаются теги для поиска.

*   **Вход:** `.json` файл, созданный на шаге `chunk`.
*   **Что происходит:**
    1.  Парсится имя файла для сбора тегов.
    2.  Для товаров в квадратных скобках выполняется поиск артикулов в каталоге.
    3.  Текст чанков кодируется в векторы через `ENCODER_SERVICE_URL`.
    4.  Данные сохраняются в Qdrant.

```bash
# Индексация одного документа для тенанта по умолчанию
kb-tool index ./chunks_json/инструкция.json

# Индексация с явным указанием тенанта
kb-tool index ./chunks_json/инструкция.json --tenant azerty

# Пакетная индексация всех .json файлов из папки
kb-tool index ./chunks_json/ --tenant azerty --batch

# Перезаписать документ, если он уже существует (по умолчанию --on-duplicate ignore)
kb-tool index ./chunks_json/инструкция.json --tenant azerty --on-duplicate replace
```

#### 4. Именование файлов: система тегов

Это самый важный раздел для пользователя. **Правильное имя файла = качественный поиск.**

Имя файла (до расширения `.json`) может содержать специальные блоки в скобках. Они обрабатываются слева направо.

##### Формат
```
[Название товара] {тег1,тег2,тег3} Базовое имя документа.json
```
Все блоки опциональны и могут комбинироваться в любом порядке, но **рекомендуется придерживаться порядка, указанного выше**, для читаемости.

##### Типы блоков

1.  **Квадратные скобки `[Название товара]` — Поиск товара в каталоге**
    *   **Назначение:** Автоматически привязать документ к конкретному товару.
    *   **Действие:** Инструмент ищет это название в каталоге товаров (через `catalog_api`) и добавляет **все артикулы** найденного товара в качестве тегов.
    *   **Пример:** Файл `[Ноутбук Azerty AZ-1406] Инструкция.json` получит теги артикулов этого ноутбука, например `["120-AZ001", "120-AZ002"]`.

2.  **Фигурные скобки `{тег1,тег2}` — Произвольные теги**
    *   **Назначение:** Добавить ключевые слова, категории, тип документа, **артикулы** (теперь артикулы — это просто теги, как и любые другие).
    *   **Действие:** Указанные слова **добавляются как теги**.
    *   **Пример:** Файл `{инструкция,электробезопасность,120-001} Меры предосторожности.json` получит теги `["инструкция", "электробезопасность", "120-001"]`.

3.  **Блоки без скобок — Базовое имя**
    *   Все, что осталось после обработки скобок, становится **именем документа** в базе данных (под ним можно будет удалить документ).

##### Примеры комбинаций

| Имя файла | Тип (`doc_type`) | Имя в БД (`document_name`) | Какие теги получает документ |
| :--- | :--- | :--- | :--- |
| `[Ноутбук Azerty] Инструкция.json` | `product` | `Инструкция` | Артикулы товара `Ноутбук Azerty` |
| `{инструкция,важно,120-001} Условия.json` | `tagged` | `Условия` | `["инструкция", "важно", "120-001"]` |
| `[Ноутбук] {инструкция} Руководство.json` | `product` | `Руководство` | Артикулы `Ноутбук` + `["инструкция"]` |
| `Политика конфиденциальности.json` | `general` | `Политика конфиденциальности` | `["general"]` (тег по умолчанию) |

> **Важно:** Специальной обработки для артикулов в круглых скобках `(...)` больше нет. Если вы хотите добавить артикул как тег, просто укажите его внутри фигурных скобок `{120-001, инструкция}`.

#### 5. Управление и мониторинг

*   **Поиск (`search`):** Проверка, как работают теги и релевантность.
    ```bash
    kb-tool search "как включить ноутбук" --tenant azerty
    kb-tool search "гарантия" --tags "важно" --tenant azerty
    ```
*   **Просмотр (`list`):** Что вообще есть в базе.
    ```bash
    kb-tool list documents --tenant azerty
    kb-tool list tags --tenant azerty   # Статистика использования тегов
    kb-tool list tag --tag инструкция --tenant azerty  # Документы с тегом
    ```
*   **Удаление (`delete`):** Чистка базы.
    ```bash
    kb-tool delete document --name Инструкция --tenant azerty
    kb-tool delete tag --tag инструкция --tenant azerty --yes  # Удалить все с тегом
    kb-tool delete tenant --tenant azerty --yes  # Полная очистка тенанта
    ```

### Документация разработчика

Этот раздел предназначен для тех, кто будет развивать `kb-tool` и RAG-систему.

#### Архитектура и место в проекте

`kb-tool` — это клиентское приложение, которое использует общие модули из `shared/` и не является микросервисом.

```
project/
├── shared/                      # Общие модули для всех компонентов
│   ├── vector_db/                # Векторная БД
│   │   ├── client.py               # Низкоуровневый клиент Qdrant (единственный, кто знает про Qdrant)
│   │   ├── document_service.py     # Сервис для индексации/удаления документов (используется kb-tool)
│   │   ├── rag_service.py          # Сервис для RAG-поиска (используется BatchRAGWorker)
│   │   ├── models.py               # Pydantic модели (Chunk, RAGSearchResult...)
│   │   └── tag_normalizer.py       # Нормализация тегов
│   ├── catalog_api.py           # Высокоуровневый API для каталога (поиск артикулов)
│   ├── catalog_client.py         # Низкоуровневый клиент Redis для каталога
│   ├── encoder_client.py         # Клиент для сервиса эмбеддингов
│   ├── doc_name_parser.py        # Парсер имени файла (логика тегов)
│   └── config.py                 # Общий конфиг
│
├── kb-tool/                      # Директория с утилитой
│   ├── cli.py                      # Точка входа, парсинг аргументов CLI
│   ├── commands/                    # Реализация команд
│   │   ├── convert.py
│   │   ├── chunk.py
│   │   ├── index.py                 # Здесь используется обновленный DocNameParser
│   │   └── ...
│   ├── core/
│   │   └── chunk_processor.py       # Ядро индексации (ChunkDocumentProcessor)
│   ├── converters/                  # Конвертеры в Markdown
│   └── llm/
│       └── chunker.py                # Чанкинг через LLM (DocumentChunker)
│
├── w_rag_b/                       # Пример микросервиса, использующего shared
│   └── app/
│       └── main.py                   # BatchRAGWorker, использует rag_service
│
└── ...
```

**Ключевой принцип:** Модули из `shared` не знают о существовании `kb-tool`. Они предоставляют интерфейсы для работы с внешними сервисами.

#### Поток обработки документа (Data Flow)

1.  **`convert`**: Использует `ConverterFactory` -> `WordConverter`/`HTMLConverter` для создания `.md`.
2.  **`chunk`**:
    *   `DocumentChunker` читает `.md`.
    *   Если текст большой (> `MAX_CHUNKER_INPUT_SIZE`), используется `RecursiveCharacterTextSplitter` из LangChain для нарезки на части **с перекрытием**. Это критически важно для сохранения контекста.
    *   Каждая часть отправляется в `DeepSeekDocProc` (LLM) для выделения чанков и тем (`topics`).
    *   Результат собирается в единый JSON.
3.  **`index`**:
    *   `run_index` собирает все `.json` файлы.
    *   Для каждого файла вызывается `DocNameParser.parse_filename()`, который возвращает структуру: `(base_name, doc_type, product_name, tags_from_name)`. **Важно:** артикулы из круглых скобок больше не парсятся.
    *   Если есть `product_name`, через `catalog_api.search_products()` ищутся артикулы товара.
    *   Формируется объект `IndexingItem` с полями: `document_name`, `chunks` (список `Chunk` моделей **без векторов**), `tags` (объединенный список уникальных тегов из `tags_from_name` и найденных артикулов).
    *   `ChunkDocumentProcessor` берет на себя финальную обработку:
        *   Кодирует тексты чанков в векторы через `encoder_client.encode_batch()`.
        *   Проверяет стратегию дубликатов.
        *   Вызывает `document_service.index_document()`, передавая подготовленные `Chunk` (уже с векторами) и финальный список `tags`.

#### Ключевые моменты для разработчика

*   **Нормализация тегов:** Все теги (из скобок, артикулы, `general`) проходят через `tag_normalizer.py` (нижний регистр, пробелы заменены на `_`). Это гарантирует, что поиск по тегу `"Инструкция по эксплуатации"` и `"инструкция_по_эксплуатации"` даст один результат.
*   **Отказ от круглых скобок:** В текущей версии парсер имен (`doc_name_parser.py`) больше не обрабатывает круглые скобки `(...)` как специальный блок для артикулов. Артикулы, как и любые другие метаданные, должны указываться внутри фигурных скобок `{...}`. Это упрощает логику и делает систему более предсказуемой.
*   **`ChunkDocumentProcessor`:** Этот класс не знает о типах документов (`product`, `general`). Его задача — взять подготовленный список тегов и список чанков, добавить векторы и отдать в `document_service`. Это разделяет ответственность: парсинг имени (сложная логика) и техническая индексация.
*   **`DocumentService` vs `RAGService`:**
    *   `DocumentService` (используется `kb-tool`) оперирует понятиями "документ", "теги". Он знает про коллекции и фильтры.
    *   `RAGService` (используется `BatchRAGWorker`) предназначен только для поиска. Он возвращает модели `RAGSearchResult`, которые всегда содержат поле `tags`. Это позволяет воркеру применять свою логику взвешивания.
*   **Стратегия поиска:** В `rag_service.py` реализована **одна коллекция на тенанта**. Поиск всегда идет по тегам с логикой **OR** (любой из переданных тегов). Это упрощает архитектуру и делает её гибкой. Взвешивание результатов по приоритетам (артикул тикета важнее, чем тег `general`) происходит на стороне воркера (`BatchRAGWorker`), который получает теги в каждом результате.
*   **Расширяемость:**
    *   **Новая векторная БД:** Нужно переписать только `shared/vector_db/client.py`, сохранив его интерфейс.
    *   **Новый источник тегов:** Если появится новый тип скобок в имени файла, логика добавляется в `DocNameParser`, а итоговый список тегов все так же передается в `IndexingItem`.
    *   **Новая LLM для чанкинга:** Замена происходит в `llm/chunker.py`, в классе `DocumentChunker`.

#### Типичные ошибки при разработке

*   **Забыть про нормализацию:** Если при индексации тег сохранился как "Инструкция", а при поиске используется "инструкция", они не совпадут. Всегда используйте `normalize_tag_set()`.
*   **Игнорировать перекрытие (overlap) при нарезке:** При использовании `split_text()` из LangChain вместо `create_documents()` перекрытие не создается, и контекст на стыках частей теряется для LLM. Это приводит к "рваным" чанкам.
*   **Путаница с `Chunk` моделью:** Модель `Chunk` из `shared.vector_db.models` содержит поле `topics`. В процессе индексации они объединяются с тегами из имени файла и сохраняются в общем списке `tags` в payload точки.
*   **Прямые вызовы `qdrant_client`:** Никто, кроме `document_service` и `rag_service`, не должен импортировать `qdrant_client` напрямую. Вся работа с Qdrant — только через них.
*   **Неправильная сигнатура `parse_filename`:** После удаления обработки круглых скобок функция возвращает кортеж из 4 элементов, а не из 5. При обновлении кода убедитесь, что все вызовы этой функции были исправлены.
